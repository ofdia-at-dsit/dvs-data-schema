# Known issues

Items that have already been identified by maintainers but are not yet fixed. Each open item will be migrated to a GitHub issue on first public push, and this file will wind down as issues move to proper tracking.

Each item names the category it would be raised under and a severity in the maintainer's view.

## Resolved

### KI-1 — `$E` fenced-block markers render as literal text on GitHub

**Resolved.** All `$E ... $E` pairs in the workspace Markdown under
`schema-1.0/gpg-45/` and `schema-1.0/gpg-44/` have been converted to
`<pre class="schema-example">` HTML blocks.

- `&nbsp;` entities are now regular spaces.
- Markdown inline links inside each block became HTML `<a>` elements so
  schema cross-references remain clickable inside the example.
- Blank-line export artefacts between each field were collapsed.

The conversion is deterministic and was applied by
[`supporting-material/scripts/convert_schema_examples.py`](supporting-material/scripts/convert_schema_examples.py).
The script is idempotent: a second run is a no-op. 30 blocks were converted
(22 in GPG 45 data model, 6 in GPG 45 data dictionary, 2 in GPG 44 data
model). The `govspeak.js` plugin in the docs-site still handles `$E` for
any future publication that reintroduces them.

### KI-4 — Cross-reference anchors inside data-model files

**Resolved.** Same-file fragment links that pointed to anchors now living
in a different file have been rewritten to their cross-file form. For
example, `[date](#date)` inside `03-data-model.md` is now
`[date](04-data-dictionary.md#date)`.

The mapping is small and explicit; same-file anchors were left alone.
26 rewrites across `03-data-model.md`. The rewrite was applied by
[`supporting-material/scripts/apply_cross_file_links.py`](supporting-material/scripts/apply_cross_file_links.py)
and is idempotent: only bare `#anchor` links that map to a different file
are touched; links that already have a file prefix are left alone, so a
second run is a no-op.

### KI-6 — Colour-coded data-model tables from the 1.3 draft

**Resolved** as an additive presentation experiment under
[`supporting-material/presentation-experiments/`](supporting-material/presentation-experiments/README.md).
Two files, one for each guide, render selected data-model tables with
colour-coded type cells:
[`gpg-45-colour-coded-model.md`](supporting-material/presentation-experiments/gpg-45-colour-coded-model.md),
[`gpg-44-colour-coded-model.md`](supporting-material/presentation-experiments/gpg-44-colour-coded-model.md).
The authoritative text under `schema-1.0/` is unchanged. The colour is
applied with inline `background-color` styles; the text in every row
still carries the type as text, so readers who cannot see colour get the
same information.

### KI-3 — `content_type` / `content_format` documentation of the upstream correction

**Resolved.** A short editor's note now sits at the foot of the
`### Attachment` section in
[`schema-1.0/gpg-45/03-data-model.md`](schema-1.0/gpg-45/03-data-model.md),
pointing readers at the 13 April 2026 GOV.UK correction and linking to the
`[publication]` entry for that date in `CHANGELOG.md`. The editor's note
is explicitly marked as a workspace navigation aid, not part of the
published schema text.

### KI-9 — First real docs-site build

**Resolved.** `npm install && npm run build` was run end-to-end from a
clean tree and completed without errors in 1.7 seconds, producing 38
rendered HTML pages plus 23 passthrough assets (`govuk-frontend` CSS, JS
and vendor images).

Verified in the rendered output:

- The `<pre class="schema-example">` blocks introduced by the KI-1
  conversion render with preserved indentation and clickable inline links.
- The cross-file `target.md#anchor` links introduced by the KI-4 rewrite
  are re-resolved by the Eleventy `mdLinkRewrite` transform to the pretty
  URL form (for example `04-data-dictionary/#date`).
- The inline `background-color` styles from the KI-6 colour-coded tables
  survive Eleventy's Markdown rendering intact.
- The GOV.UK header, phase banner, grid, content columns and footer all
  render with the correct `govuk-*` classes applied by the `govspeak.js`
  plugin.
- `govuk-frontend.min.css` is written to `_site/stylesheets/` with
  `url(/assets/...)` references intact when `BASEURL` is empty, and
  rewritten to `url($BASEURL/assets/...)` when set.

A `package-lock.json` was generated by `npm install` and committed so CI
can use `npm ci` for reproducible installs.

## Open

### KI-2 — Kramdown `{:#anchor}` references in source, not used in workspace

**Category:** structural-feedback
**Severity:** very low

**Files affected:** none currently — the cleaned Markdown source provided for this repo appears not to use Kramdown-style anchor attributes. This item exists as a reminder: if a future schema publication reintroduces them, apply the same fix used in the trust framework repo (convert `{:#section-N}` to invisible `<a id="section-N"></a>` anchors).

### KI-5 — CODEOWNERS handles are placeholders

**Category:** structural-feedback
**Severity:** blocker for public push
**Files affected:** [`.github/CODEOWNERS`](.github/CODEOWNERS), any issue templates that reference `OWNER/REPO` in links.

**What is wrong:** Team handles `@ofdia-maintainers`, `@ofdia-policy`, `@ofdia-engineering` and the string `OWNER/REPO` are placeholders. These need to be replaced with the real GitHub organisation and team handles before the repository is used publicly, otherwise review routing will silently fail.

**Suggested fix:** a maintainer with access to the organisation's GitHub settings edits `.github/CODEOWNERS` and the relevant template config files directly.

### KI-7 — Drafting clarifications carried over from internal comments

**Category:** schema-feedback
**Severity:** low (not actionable in this repo, record-only)

**What is wrong:** Internal drafting comments raised recurring clarification questions about whether references should use the revised trust framework name, whether wording should use "supplementary code", what some organisation references point to, whether some fields support multiple trust frameworks, and whether certain fields might be renamed or grouped differently. These are upstream drafting points, not repository cleanup items.

**Suggested fix:** none in this repository. Track as `schema-feedback` issues so the notes persist for future GOV.UK revisions. The
[`supporting-material/mappings/`](supporting-material/mappings/README.md)
folder now also hosts the OIDC / OID4IDA / VC alignment conversation in
concrete form and can absorb specific field-rename proposals as mapping
annotations.

### KI-8 — International alignment notes

**Category:** schema-feedback
**Severity:** record-only

**What is wrong:** Internal notes record OIDF feedback on the 1.0 schema (alignment with OID4IDA, IANA/OIDC-style claim names, age-assertion wording, the meaning of "token"), and also record pushback on some of those suggestions. This context is relevant for future publication revisions but is not a change the repository can make unilaterally.

**Suggested fix:** none in this repository. Track as `schema-feedback` issues. Concrete alignment gaps are now documented as `target_only` and `restructured` entries in
[`supporting-material/mappings/oid4ida-mapping.yaml`](supporting-material/mappings/oid4ida-mapping.yaml),
so future discussion has a specific field-level vocabulary to reference.
